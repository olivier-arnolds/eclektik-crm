## The three you reported — root cause + fix

### 1. "Some places I can't tag SEER"
Root cause: the **inline deal panel** (`InlineDealDetail` in `inline-details.jsx`) — the
one you actually edit deals in — had a Currency picker but **no product-line field at
all**. The "Seer / Glint / Insights" picker only existed in the separate deal-detail
*modal*, which you rarely open. So there was simply nowhere to set Seer inline.

Fixed: added a **Product line** dropdown next to Currency in the inline panel, with
Glint / ROI / Seer / Insights / Other. If a deal already holds a non-standard value
(e.g. "Glint Consultancy Services") it's preserved as an option rather than wiped.

### 2. "If I look at a company I can't scroll down"
Root cause: the expandable account **core-details block** (address, website, industry
dropdown, parent account…) was rendered inside the **fixed hero strip**, which sits
*above* the scroll area and is clipped (`overflow:hidden`). When expanded it overflowed
the hero and the lower fields — including the industry dropdown — were unreachable.

Fixed: moved that block into the scrollable region (`.acc-scroll`) in
`lane-accounts.jsx`. The hero stays compact; all detail now scrolls.

### 3. "Connecting to Microsoft takes many clicks before I see content"
Root cause: `fetchGraphData()` in `BDApp.jsx` returns early if there's no `graph_token`,
and its effect only re-ran on **session change** — not when the token actually arrived.
So right after connecting, the token was present but no fetch was triggered; mail/Teams/
calendar only appeared after some *other* action forced a refetch (navigating, the
10-minute auto-refresh, etc.).

Fixed: the fetch now also re-runs the moment `hasGraphToken` flips true, so content
loads immediately after connecting — no extra clicks.

## Also fixed now (low-risk quick wins from the sweep)

- **Tasks tab could hang on "Loading…" forever.** The Supabase fetch in `tasks-view.jsx`
  had no `.catch()`, so a network failure left the spinner stuck. Added error handling
  that always clears loading.
- **Silent Teams / LinkedIn thread failures.** In `lane-comms.jsx` a failed thread fetch
  silently showed "No messages in this thread", indistinguishable from a genuinely empty
  chat. Now shows a clear error ("Could not load this Teams thread. Microsoft may need
  reconnecting.") so you know to reconnect rather than assume it's empty.

## Recommended next (not changed yet — your call)

High value, low effort:
1. **Centralise the product-line list.** `['Glint','ROI','Seer','Insights','Other']` is
   hard-coded in ~6 files. One shared constant prevents them drifting out of sync (e.g. a
   future "Seer" rename only changes one place).
2. **Reset cross-view state on tab switch** (`BDApp.jsx`). Open contact/deal context can
   linger when you jump between Workspace / Tasks / Marketing. A small effect that clears
   it on `activeView` change tightens this up.
3. **Keyboard access on clickable table rows** (reporting, marketing). Rows use `onClick`
   on `<tr>` with no `role`/keyboard handler — fine with a mouse, invisible to keyboard.

Medium:
4. **Microsoft token resilience.** Supabase doesn't persist the Graph `provider_token`
   across refreshes, so the token silently disappears and you have to reconnect. Options:
   show a clear "Microsoft disconnected — reconnect" banner the moment it's gone, and/or
   a token-refresh path. This is the deeper cause behind the reconnect friction.
5. **Lead-promote partial failures** (`lead-promote.js`). Reparenting child rows is
   best-effort with no transaction; a mid-way failure can orphan rows. Worth tightening
   the error handling / logging.
6. **Inline-edit save feedback.** Marketing email + several inline fields save on blur
   with no success confirmation; add a brief flash/toast so saves feel acknowledged.

Lower:
7. Make `TARGET_Q` / `MIN_Q` (reporting targets) DB-driven instead of hard-coded.
8. Wire up or remove the unused `onOpenTweaks` settings button in the Topbar.
9. Empty-swimlane affordance ("drag here") in the funnel.
10. FX-rate fetch failure is silent — log it and optionally surface a "rates unavailable,
    showing 1:1" note in Reporting.

# Round 2 — full-coverage sweep (the modules round 1 skipped)

Covered: War room / journey board, Calendar, Playbooks + the playbook execution API,
the app shell, auth/login, every `/api/*` serverless endpoint, responsive/mobile,
accessibility, performance, and data integrity.

## Also fixed now (v1.34.6 — safe, high-value)

- **No error boundary → whole-app white screen.** One uncaught render error blanked the
  entire app. Added a top-level `ErrorBoundary` (in `main.jsx`) with a friendly message
  and a Reload button.
- **App could hang on "Loading…" forever.** `usePipelineData.fetchAll` had no try/catch —
  a single failed Supabase query left the spinner stuck permanently. Wrapped it so loading
  always clears and the error is logged.
- **Journey drag could silently desync from the DB.** `moveJourney` updated the card
  optimistically and swallowed any save error, so a failed move looked successful until
  reload. Now it rolls the card back to its previous lane if the save fails.
- **Calendar task drag failed silently.** `moveTaskToDate` had no error handling. Now it
  surfaces the error instead of pretending the move worked.
- **Journey search "no results" was a blank board.** Added a clear "No projects match …"
  message.

## Bigger items found — NOT quick wins, need your go-ahead

These are real and matter for "a top app," but each is a project (and risky to do blind
without being able to run the app), so I left them for you to greenlight:

1. **API auth hardening (HIGH / security).** Several serverless endpoints
   (`account-summary`, `company-insights`, `lusha`, `glint-sync`, `marketing-send`, the
   `unipile` write actions, `admin-weekly-export`) use the service key and have **no auth
   check** — anyone who knows the URL could call them (read account briefs, trigger
   Lusha email lookups, overwrite glint_delivery, send campaigns). Webhooks are fine
   (`marketing-webhook` verifies a Svix signature). Recommend a shared guard that
   verifies the caller's Supabase JWT (or a shared secret) on every non-webhook,
   non-cron endpoint. Also: validate `redirect_url` in `unipile.js` against your domains.
   The good news: no secrets are leaked and no stack traces escape to clients.
2. **Mobile / tablet unusable (HIGH).** `styles.css` has **zero `@media` queries**. The
   3-lane workspace needs ~800px; below that, panes wrap and modals clip. If anyone
   opens the CRM on a phone/tablet it's broken. Needs a responsive pass (stack lanes,
   full-width modals, collapse the calendar).
3. **Performance as data grows (MEDIUM).** `usePipelineData` fetches ~7k rows in one
   blocking batch on load (3–5s on slower links); `adaptDeal`/`adaptContact` do O(N²)
   array `.find()` lookups (500 deals × 1000 accounts). Pre-indexing accounts/contacts
   into Maps and paginating would keep it fast at scale. The `comms` query also fetches
   1000 rows already correctly ordered — fine.
4. **Accessibility (MEDIUM).** Clickable `<div>`/`<span>`/`<tr>` lack keyboard support;
   modals don't close on Esc or trap focus; some inputs have no label. Fine with a mouse,
   blocked for keyboard/screen-reader users.
5. **Calendar timezone edge (MEDIUM).** Task due-dates are parsed with `new Date("YYYY-MM-DD")`
   (UTC midnight) but bucketed by local day — can land a task on the wrong day for some
   timezones. (Low impact at UTC+1/+2, but worth fixing with a local-date parse.)
6. **Maintainability (MEDIUM).** `['Glint','ROI','Seer','Insights','Other']`, the owner
   map, and stage metadata are each duplicated across several files and can drift.
   Centralise into one `constants.js`.
7. **Playbook drafts (MEDIUM).** AI-draft generation can insert an empty body on model
   failure, and the no-em-dash content rules aren't re-validated after generation. Add an
   empty-body guard + a post-generation rule check.

## What a true tester pass still needs
I reviewed code, which finds logic/structure bugs. It can't catch runtime, visual,
cross-browser or responsive issues that only appear in the running app. The next level is
driving the live CRM interactively (I can do that via the Chrome connection, with you
logged in) to click every workflow and watch what actually breaks.

## Verdict
No data loss and no leaked secrets. The headline risks are the **unauthenticated API
endpoints** and **no mobile support** — both worth scheduling deliberately. Day-to-day the
app is solid; the round-1 and round-2 fixes remove the real friction and the
crash/hang/silent-failure traps. The numbered list above is the roadmap from "works well"
to "top-tier."
